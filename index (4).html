Lab · Microsoft Entra ID

Entra ID OIDC onboarding & incident runbooks

Impact

Configured OIDC sign-in end-to-end on Microsoft Entra ID, hit two real authentication incidents, and wrote audit-ready runbooks — control implementation with the evidence a real audit expects.

What I did

Built OIDC sign-in with MSAL, then debugged two live failures from the logs: AADSTS50011 (a redirect-URI mismatch caused by a single trailing slash) and AADSTS7000215 (the secret ID copied instead of the secret Value). Corrected the App Registration, rotated the secret, and documented runbooks RB-001 and RB-002.

Concept proven

Entra ID is exact — one character breaks the whole auth flow, and Sign-in & Audit logs are where you find the truth. Mapped to NIST CSF and ITIL incident/problem/change.

Lab · Entra ID / testing

Entra ID OIDC token test framework

Impact

Took the pytest foundation from the previous lab and aimed it at a live Microsoft Entra ID OIDC app registration — testing a real enterprise identity platform.

What I did

Wrote 29 passing tests checking token-endpoint status codes, JWT response structure, response time, rejection of invalid credentials, blocking of unregistered redirect URIs, and real token acquisition via the MSAL client-credentials flow. Every run showed up in Entra ID Sign-in logs.

Concept proven

Entra ID doesn’t reject an unregistered redirect URI immediately — it shows the login page first and blocks only after authentication. The kind of detail that catches SSO integrations at scale.

Lab · Authorization / RBAC

Role-Based Access Control with Auth0

Impact

Proved the control that stops a logged-in user from reaching admin-only resources — the safeguard behind privilege-misuse prevention and a core requirement in most security audits.

What I did

Registered an OIDC application in Auth0, created admin and user roles, provisioned two users, and wrote a Post-Login Action to inject roles into the token as a custom claim. Built a single-page app that reads the role and enforces access, returning “Access Denied” for unauthorized users.

Concept proven

Authentication (who you are) vs authorization (what you are allowed to do). Being logged in is not the same as being allowed in.

What the labs taught me

Patterns I keep seeing

Across every lab, the same kinds of problems came up again and again. These are the lessons I now carry into any identity or integration work.

The tool says everything is fine when it is not

Many failures were silent — no error, just wrong or missing data. I learned to never trust a green tick on its own and to always check the real output: row counts, token validity, and what actually reached the destination.

Authentication is the hardest part every time

Tokens, headers, redirect URIs and OAuth settings caused more trouble than the actual logic. The small details mattered most: one wrong character in a token, a missing Authorization header, or a redirect URI that did not match exactly.

Error messages often point the wrong way

Generic messages like “please contact support”, or a raw code error shown to the user, hid the real cause. I learned to read past the message and find the true root cause in the logs.

Date and format mismatches break things quietly

Joins returned nothing and comparisons failed because two systems stored the same value in different formats. Aligning the formats before comparing fixed most of these.

Go to the source documentation

Most fixes did not come from a platform help page. They came from the original vendor docs, the API guide, or testing the actual behaviour myself.
