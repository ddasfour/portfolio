Lab · Microsoft Entra ID

Entra ID OIDC onboarding and incident runbooks

Impact

I set up OIDC sign-in on Microsoft Entra ID from start to finish. I hit two real login errors, fixed them, and wrote clear runbooks with the evidence an audit needs.

What I did

I built OIDC sign-in with MSAL, then fixed two live errors using the logs. AADSTS50011 was a redirect URI that did not match, caused by one extra slash. AADSTS7000215 happened because I copied the secret ID instead of the secret Value. I corrected the App Registration, rotated the secret, and wrote runbooks RB-001 and RB-002.

What it proves

Entra ID is strict. One wrong character can break the whole login. The Sign-in and Audit logs show what really went wrong. I mapped the work to NIST CSF and ITIL.

Lab · PowerShell / IAM JML

PowerShell inactive-account tool

Impact

I built a PowerShell tool that finds dormant accounts. Old, unused accounts are a common security risk, and this tool supports the leaver step of the joiner, mover, leaver process.

What I did

I wrote a script that finds users who have not logged in for 90 or more days. It exports the list to a CSV file and flags the accounts to review before they are turned off. I also fixed execution-policy blocks and Git merge conflicts on the way.

What it proves

Unused accounts are an easy target. Finding them automatically is the first step to closing the gap in the leaver process.

Lab · Entra ID / testing

Entra ID OIDC token test framework

Impact

I took the pytest base from the last lab and pointed it at a live Microsoft Entra ID app. This time the target was a real enterprise platform.

What I did

I wrote 29 tests, and they all passed. They check token status codes, JWT structure, response time, rejection of bad credentials, blocking of unregistered redirect URIs, and getting a real token with the MSAL client-credentials flow. Every run showed up in the Entra ID Sign-in logs.

What it proves

Entra ID does not block an unregistered redirect URI right away. It shows the login page first and blocks only after login. This small detail can catch teams out when building SSO.

Lab · Authorization / RBAC

Role-Based Access Control with Auth0

Impact

I showed the control that stops a logged-in user from reaching admin pages. This is a basic safeguard and a common audit requirement.

What I did

I set up an app in Auth0 with two roles, admin and user, and two users. I added a Post-Login Action to put each user's role into their token. Then I built a small app with four pages that checks the role and shows “Access Denied” to users who are not allowed.

What it proves

Authentication is who you are. Authorization is what you can do. Being logged in is not the same as being allowed in.

Lab · Federation / SSO

SAML 2.0 Single Sign-On with Auth0

Impact

I showed how enterprise SSO shares identity between an Identity Provider and a Service Provider. SAML still runs a lot of large enterprise apps.

What I did

OpenID Connect with Auth0

Impact

I showed the modern standard used by web and mobile apps, and compared it with SAML.

What the labs taught me

Patterns I keep seeing

The same kinds of problems came up again and again. These are the lessons I now carry into any identity or integration work.

The tool says everything is fine when it is not

Many failures were silent. There was no error, just wrong or missing data. I learned not to trust a green tick on its own. I now always check the real output: row counts, token validity, and what actually reached the destination.

Authentication is the hardest part every time

Tokens, headers, redirect URIs and OAuth settings caused more trouble than the logic. The small details mattered most: one wrong character in a token, a missing Authorization header, or a redirect URI that did not match exactly.

Error messages often point the wrong way

Generic messages like “please contact support”, or a raw code error shown to the user, hid the real cause. I learned to read past the message and find the true root cause in the logs.

Date and format mismatches break things quietly

Joins returned nothing and comparisons failed because two systems stored the same value in different formats. Fixing the formats before comparing solved most of these.

Go to the source documentation

Most fixes did not come from a help page. They came from the vendor docs, the API guide, or testing the real behaviour myself.
